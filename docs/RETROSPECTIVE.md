## 二、出现的问题及根因

| 问题 | 根因 | 教训 |
|------|------|------|
| `ModuleNotFoundError: No module named 'engine'` | Python 导入路径未设置 | 设计文档应明确说明入口脚本的路径配置 |
| 插件执行路径错误 `"...\backend\nodes\filter\backend\..."` | 相对路径拼接 + cwd 设置冲突 | 应统一使用绝对路径，设计文档应明确路径策略 |
| `sourceHandle` vs `source_handle` | 前后端字段命名不一致 | 应在设计阶段明确 API 的 JSON 格式规范 |
| output 节点收到 0 items | `_gather_inputs` 未正确映射 handle | 数据流转设计应有更明确的规范 |
| 测试数据放错位置 | 独立节点没有数据源概念 | 设计时应明确"数据源节点"的职责 |

## 三、流程改进建议

### 1. 增加"模块独立测试"环节
```
实现 models.py → 写测试用例验证序列化/反序列化
实现 graph.py → 写测试用例验证拓扑排序
...
```
**好处**：问题早期发现，避免集成后排查困难

### 2. 设计阶段明确"技术约束"
- **路径策略**：始终使用绝对路径
- **API 格式**：明确 JSON 字段命名（驼峰/下划线）
- **错误处理**：明确哪些异常需要暴露给前端

### 3. 建立"冒烟测试"清单
每次修改后必测：
- [ ] 插件扫描是否正常
- [ ] 单节点执行是否正常
- [ ] 两节点串联数据是否正确传递

### 4. 实现自检清单
- [ ] 每个模块实现后立即写测试用例
- [ ] 测试用例覆盖正常路径和异常路径
- [ ] 用例全部通过后再进入下一模块

## 六、核心问题总结

### 设计文档与实现的"空白地带"

设计文档描述了**做什么**，但没有明确**怎么做**的技术约束：

1. **路径策略不明确**
   - 哪些路径用绝对路径，哪些用相对路径
   - 执行时的工作目录 (cwd) 如何设置
   - 插件路径如何存储和传递

2. **API 格式规范缺失**
   - JSON 字段用驼峰还是下划线
   - 日期时间格式
   - 错误响应结构

3. **数据流转机制模糊**
   - 节点间数据如何传递
   - handle 映射的具体实现
   - 多输入/多输出如何处理

## 九、前端实现避坑指南

### 1. 实现前：明确接口契约

| 后端教训 | 前端预防措施 |
|---------|-------------|
| `sourceHandle` vs `source_handle` | 前后端定义**统一的 TypeScript 接口**，双方签字确认 |

```typescript
// types/api.ts - 前后端共享契约
interface EdgeData {
  id: string;
  source: string;
  target: string;
  sourceHandle: string;  // 统一驼峰
  targetHandle: string;  // 统一驼峰
}
```

### 2. 实现中：模块测试 + 冒烟测试

| 阶段 | 动作 |
|------|------|
| 完成 API 服务层 | 手动调用接口验证连通性 |
| 完成 Hook | 验证状态更新 |
| 完成组件 | 验证渲染和交互 |
| 完成数据流 | 验证节点拖拽、连线、配置 |

### 3. 关键检查点

```
[ ] API 服务层 - 能获取节点列表吗？
[ ] React Flow 集成 - 能显示节点吗？
[ ] 拖拽功能 - 能从工具箱拖到画布吗？
[ ] 连线功能 - 两个节点能连线吗？
[ ] 配置面板 - 点击节点显示配置吗？
[ ] 执行流程 - 点运行能看到状态变化吗？
```
